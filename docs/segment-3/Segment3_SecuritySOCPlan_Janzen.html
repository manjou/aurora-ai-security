Threat Descriptions (Summary)

Threat 1 — Social Engineering / Phishing

Targeted phishing against consultants or executives (whaling — CEO Alex Brandt, CFO Jordan) steals credentials. Attacker accesses VLAN 40 client audit data. With victim's cross-VLAN permissions, lateral movement to VLAN 20 (HR) or VLAN 50 (AI Lab) is possible. Client confidentiality breach → GDPR violation → reputational damage for a trust-based consultancy. Mitigations: MFA on all accounts (NIS2 mandate), email filtering, least-privilege ACLs, Wazuh anomalous login alerts.

Threat 2 — Dirty Lab Sandbox Escape

VLAN 51 hosts active exploit tools and client AI models under test. An attacker or escaped exploit tool accesses client AI training data or tampers with red team results — Aurora delivers a falsely "clean" audit report. Mitigations: Strict egress filtering on VLAN 51, Wazuh FIM monitoring, separate Dirty Lab credentials.

Threat 3 — DDoS on IPsec VPN

Volumetric DDoS saturates the Basel–Freiburg IPsec tunnel (FortiGate 40F HA pair). Basel consultants lose access mid-engagement → SLA breach → revenue loss. AI-backed DDoS attacks are increasingly sophisticated. Mitigations: FortiGate HA, ISP scrubbing, LTE failover for Basel.

Threat 4 — Client Data Theft via DMZ Portal

Attacker exploits vulnerability or stolen credentials on VLAN 100 client portal to access confidential audit reports. Reports contain detailed client infrastructure vulnerabilities — Aurora becomes the supply chain attack vector. Mitigations: WAF, portal MFA, regular patching, penetration testing.

Threat 5 — Log Deletion After Privilege Escalation

Attacker with compromised admin account deletes SIEM logs to cover tracks — violating NIS2 6-month retention (Art. 23) and destroying forensic evidence. A security consultancy that cannot investigate its own breach suffers catastrophic reputational damage. Mitigations: Immutable NAS backup, Wazuh FIM on log directories, HashiCorp Vault for time-limited admin credentials.

2 Security Framework Justification — NIST CSF 2.0

As outlined in our Segment 1 network design, Aurora AI Security GmbH selected NIST CSF 2.0 over ISO 27001 and MITRE ATT&CK as the foundational security framework. As a startup-stage consultancy, CSF provides the right balance of structure and flexibility, with ISO 27001 certification as a 3–5 year goal.

Why CSF over ISO 27001

NIST CSF 2.0 provides Aurora with a flexible framework for internal risk management. The cost and operational overhead of ISO 27001 certification is not yet justified at our current stage. CSF gives structural guidance to build a mature security program first. Critically, CSF is framework-agnostic and maps directly to ISO 27001 controls — implementing CSF now does not block certification later.

Why CSF over MITRE ATT&CK Alone

MITRE ATT&CK focuses exclusively on adversary tactics and techniques at a technical level — valuable for SOC detection but it does not address governance, compliance, or business risk management. NIST CSF covers the full lifecycle from governance through recovery. MITRE ATT&CK is used as a complementary tool within the Detect and Respond functions of CSF.

4 Device Hardening Plan

Device hardening reduces the attack surface by disabling unused services, enforcing strong authentication, and restricting management access. Two layers are documented: lab simulation (pfSense, Alpine, Wazuh, Win11) and production (Cisco Catalyst, FortiGate).

7 Incident Response Playbook

7.1 Standard IR Phases

PhaseDescriptionResponsible
1. DetectionWazuh alert — analyst confirms real incident vs. false positiveTier 1/2
2. ContainmentStop the spread — isolate system, revoke compromised sessions/credentialsTier 1/2 + Tier 3
3. EradicationRemove threat — assess blast radius, close attack vectorTier 3
4. RecoveryRestore normal operations — verify clean state, return user to networkTier 1/2 + Tier 3
5. Lessons LearnedPost-incident report (5 days) — root cause, NIS2 obligations, rule updatesSOC Manager + Tier 3
Evidence Preservation: Before any containment action, snapshot logs and system state. Modifying a system before preserving evidence destroys forensic value and may violate NIS2 reporting requirements.

7.2 Playbook Summary — All 5 Threats

Playbook 1 — Phishing / Social Engineering

Detection: Wazuh auth alert — multiple failed logins or anomalous login location/time.
Containment: Revoke active sessions, disable account, isolate workstation to quarantine VLAN, block source IP at FortiGate.
Eradication: Reset credentials + MFA token. Assess blast radius — what did attacker access? Scan for keyloggers/malware.
Recovery: Re-image if malware found. Restore access with new credentials. Monitor account 2 weeks post-incident.
Lessons Learned: Assess GDPR/NIS2 obligation. Update Wazuh detection rules. Schedule phishing awareness training.

Playbook 2 — Dirty Lab Sandbox Escape

Detection: Wazuh alert — unexpected VLAN 51 outbound or lateral movement attempt.
Containment: Full VLAN 51 network shutdown at FortiGate. Preserve all logs immediately.
Eradication: Identify escaped tool/compromised node. Assess blast radius — client AI data exfiltrated?
Recovery: Restore Dirty Lab from known-good snapshot. Re-enable VLAN 51 only after full investigation.
Lessons Learned: Mandatory NIS2 notification if client data involved. Tighten VLAN 51 egress rules.

Playbook 3 — DDoS on IPsec VPN

Detection: FortiGate tunnel-down alert or Basel Wazuh agents disconnect simultaneously.
Containment: FortiGate HA failover. Contact ISP for upstream scrubbing. Activate LTE failover for Basel.
Eradication: External attack — depends on ISP scrubbing. Apply rate limiting on VPN gateway.
Recovery: Restore IPsec tunnel. Verify Basel branch connectivity. Document downtime for SLA review.
Lessons Learned: Calculate business impact. Review ISP DDoS protection contract.

Playbook 4 — Client Data Theft via DMZ Portal

Detection: WAF alert — auth failures or unusual data download from VLAN 100.
Containment: Disable compromised account, block source IP, take portal offline if active exploitation confirmed.
Eradication: Patch vulnerability. Assess blast radius — which client reports accessed? Reset all portal credentials.
Recovery: Security-test portal before bringing back online. Implement client MFA if not active.
Lessons Learned: GDPR Art. 33 notification if EU client PII accessed (72h). Commission portal penetration test.

Playbook 5 — Log Deletion After Privilege Escalation

Detection: Wazuh FIM alert — modification/deletion in /var/ossec/logs/ or wazuh-manager stopped.
Containment: Revoke admin account, isolate Wazuh server, secure immutable NAS backup as primary evidence.
Eradication: Trace original breach. Use NAS backup logs to reconstruct full timeline and attacker actions.
Recovery: Restore deleted logs from NAS. Rebuild Wazuh if compromised. Reset all admin credentials.
Lessons Learned: NIS2 mandatory notification. Implement HashiCorp Vault for time-limited admin credentials.
